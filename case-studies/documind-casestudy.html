Design Decisions

Why these choices matter.

Every architectural choice was made to solve a concrete problem, not to sound impressive.

DECISION 01
Semantic chunking over fixed windows

Fixed token windows cut sentences mid-thought, destroying context. Semantic chunking with overlap preserves meaning at boundaries — retrieval quality is noticeably better on longer documents.

Result: measurably fewer retrieval misses on multi-page documents.
DECISION 02
all-MiniLM-L6-v2 over larger models

A 384-dim model that runs fast locally with no API cost. For document retrieval at this scale, it matches much larger embedding models — choosing the right tool, not the most impressive-sounding one.

Result: zero embedding API cost, sub-second latency per chunk.
DECISION 03
ChromaDB over in-memory vectors

ChromaDB persists embeddings to disk — re-querying the same document doesn't require re-processing. Critical for usability when documents take seconds to embed.

Result: re-query latency drops from seconds to milliseconds.
DECISION 04
Classification before extraction

Knowing a document is a resume vs. a contract changes what you extract. Classification lets the prompt be document-type-aware — improving structured output quality significantly across all 6 document classes.

Result: extraction prompts are 60% more specific post-classification.
DECISION 05
Streamlit over Gradio

Gradio is excellent for rapid ML prototyping, but Streamlit unlocks a proper application architecture: session state, multi-tab layout, sidebar controls, and native dark theming that matches the pipeline's production intent.

Result: demo-to-product feel, deployable to Streamlit Cloud in one push.
DECISION 06
Source citation UI was unplanned

The chunk citation panel wasn't in the original spec. Added after noticing that users questioned AI answers without evidence. Showing retrieved context chunks directly in the UI transformed trust in the system.

Result: transparency is UX. The most-praised feature in every demo.

Learnings

What building this actually taught me.

The gap between "it works in a notebook" and "it works on real documents" is where the real engineering happens.

01
RAG quality lives or dies by chunking

Naive splitting produced retrieval misses that no amount of prompt engineering could fix. The fix was upstream — semantic chunking with overlap solved what the LLM layer couldn't.

02
OCR preprocessing is non-trivial

Deskewing and denoising before pytesseract significantly improved text quality on real-world scanned documents. Garbage in, garbage out applies hard when documents are the data source.

03
Showing retrieved chunks builds trust

The source citation UI wasn't in the original spec. Added after noticing users questioned AI answers without evidence. Transparency isn't a feature — it's foundational to how people relate to AI output.

04
Distribution matters as much as intelligence

The email output was the highest-leverage feature in demos. Stakeholders want summaries in their inbox, not dashboards they have to log into. Building for where people already are beats building the best interface.

05
Modular architecture compounds

Because each pipeline layer is independent, migrating the frontend from Gradio to Streamlit took a day, not a week. The RAG pipeline attached to the existing extraction layer in two days. Design for replaceability.

06
UI framework choice shapes perceived quality

The Streamlit migration didn't change the backend at all — but the polished multi-tab dashboard with live pipeline metrics changed how every demo recipient perceived the product. Interface is product.
